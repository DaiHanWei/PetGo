### 🔐 用户登录与引导（User Auth & Onboarding）
> 对应 PRD §4.0，FR-0A 至 FR-0H。如需修改登录功能，以本节为起点。

#### 核心决策记录

| 决策项 | V1 方案 | 决策原因 | V2 规划 |
|--------|--------|---------|---------|
| 登录方式 | **仅 Google 登录** | 降低 V1 接入复杂度，Google 在印尼年轻用户中覆盖广 | 手机号+OTP（WhatsApp/SMS）、Apple 登录 |
| 登录时机 | **先浏览，触发功能时登录** | 降低冷启动拦截率，让用户先感受内容价值 | — |
| 注册/登录入口 | **合并**（Google 首次授权=自动注册）| 减少用户判断成本 | — |

#### 完整引导流程

```
App 启动 → Splash → 首页（未登录可浏览）
                          │
              ┌───────────┴────────────────────────┐
              │ 浏览到第3页时（FR-0B）               │
              ↓                                    │
         软性登录推荐浮层                            │
         主CTA显著 / 关闭按钮小且颜色浅              │
         关闭后：本次 session 不再弹                 │
              │                                    │
              └────────────────────────────────────┘
                          │
         触发核心功能时（FR-0C）
         问诊 / 发布 / 档案
                          ↓
                  强引导登录弹窗
                          ↓
                   Google 登录（FR-0D）
                          │
              ┌───────────┴──────────────┐
            老用户                      新用户
         回到触发点继续                     │
                               Step 1：昵称确认（FR-0E）
                               默认 Google 账户名，可修改
                                          ↓
                               Step 2：宠物状态选择（FR-0F，必选）
                          ┌──────────────┼──────────────┐
                          A              B               C
                      有宠物          计划养宠          爱好者
                          ↓              ↓               ↓
                    档案创建引导      直接进首页        直接进首页
                    （FR-0G）        无提示条          无提示条
                    可跳过
                    ┌──────────┐
                  完成        跳过
                    ↓           ↓
                  首页       首页+提示条（FR-0H）
```

#### 首页提示条逻辑（FR-0H）

| 项目 | 规则 |
|------|------|
| 触发对象 | 选了「有宠物」(A) 但未完成档案上传的用户 |
| 显示位置 | 首页顶部固定区域 |
| 关闭行为 | 当次 session 关闭后不再显示，下次重启继续 |
| 频次上限 | **前3次重启均显示**，第3次关闭后永不再显示 |
| 自动停止 | 完成档案上传后立即停止 |
| 不触发 | 选 B（计划养）或 C（爱好者）的用户 |

#### 宠物状态三分类说明

| 状态 | 标签 | 首页内容方向 | 档案提示条 |
|------|------|------------|---------|
| A | 我有宠物，上传宠物档案 | 健康/问诊/档案为主 | 有（跳过建档时） |
| B | 暂无宠物，但计划养 | 养宠知识/选宠指南为主 | 无 |
| C | 宠物爱好者，无养宠计划 | 宠物内容/社区为主 | 无 |

> 用户可在「个人设置 → 宠物状态」中随时修改分类。

- [ ] 新增登录方式（如手机号 OTP）→ 更新 FR-0D，新增对应 FR
- [ ] 调整软性弹窗触发页数 → 修改 FR-0B
- [ ] 调整提示条显示次数 → 修改 FR-0H 频次规则

> 对应 PRD §4.4，FR-17 至 FR-18。如需修改首页内容逻辑，以本节为起点。

#### 核心决策记录

| 决策项 | V1 方案 | 决策原因 | V2 规划 |
|--------|--------|---------|---------|
| 内容来源 | **三类全进公共 Feed**（日常分享 + 专业科普 + 成长日历）| Feed 是核心内容场，需要足够的内容密度 | — |
| 排序逻辑 | **时间倒序**（纯时间，无算法）| V1 工程量可控，数据量不够算法也无意义 | 算法推荐权重 |
| 状态过滤 | **硬过滤**（按宠物状态 A/B/C）| 提升内容相关性，减少不相关内容干扰 | — |
| 未登录首页 | **显示全部内容，不过滤** | 降低冷启动门槛，让游客感受内容价值 | — |
| 冷启动方案 | **种子内容 + 引导发布** | 避免新用户打开看到空页面流失 | — |

#### 宠物状态过滤规则（硬过滤，V1）

| 用户状态 | 显示内容类型 |
|---------|------------|
| A（有宠物）| 三类全显示 |
| B（计划养宠）| 专业知识科普 + 日常分享（**不显示**成长日历快乐时刻）|
| C（宠物爱好者）| 三类全显示 |
| 未登录 | 三类全显示，无过滤 |

#### 核心决策记录

| 决策项 | V1 方案 | 决策原因 |
|--------|--------|---------|
| 导航类型 | **底部 Tab Bar，5个位置** | 行业标准，符合用户习惯 |

#### Tab Bar 结构

```
首页  |  成长档案  |  [+]  |  问诊  |  我的
 ✅         ❌        ❌       ❌       ❌
（无需登录） （需要登录）
```

| Tab | 未登录 | 内容说明 |
|-----|--------|---------|
| 首页 | ✅ 可访问 | 公共 Feed，按宠物状态过滤 |

### 👤 个人中心（我的 Tab）
> 对应 PRD §4.6，FR-20 至 FR-21。如需修改个人中心，以本节为起点。

#### 页面结构决策

| 区块 | 包含 | 备注 |
|------|------|------|
| 用户信息 | 头像 + 昵称 + 编辑按钮 | 头像默认 Google 头像，可替换 |
| 宠物状态 | 当前状态展示 + 修改入口 | **双入口**：我的 + 成长档案（FR-21）|
| 我的发布 | 全部已发布内容列表 | 三类内容均在此汇聚 |
| 退出登录 | 一键退出 | 返回首页游客状态 |
| 账号注销 | 删除账号与数据 | **PDP 合规强制要求**，双重确认 |

#### 关键设计决策

- **宠物状态双入口**：「我的」和「成长档案」都能改，效果同步，复用 FR-0F 选择界面

- **退出登录**：清除本地 token，返回首页游客状态，不删数据

### 🔔 推送通知（Push Notifications）
> 对应 PRD §4.7，FR-22A 至 FR-22D。如需修改通知逻辑，以本节为起点。

#### 通知场景决策

| 通知场景 | 类型 | V1 是否做 | 说明 |
|---------|------|---------|------|
| 兽医回复 | 系统推送 | ✅ | 核心体验，必须做 |
| 被点赞 | 系统推送 | ✅ | V1 有点赞功能 |
| 被评论 | 系统推送 | ✅ | V1 有评论功能 |
| App 版本更新 | **App 内提示** | ✅ | 不用推送权限，冷启动检测 |
| 收藏/@ 提及 | — | ❌ | V1 不做收藏和@ |

#### 关键设计决策

- **推送权限申请时机**：完成第一次问诊后触发，不在冷启动时要求（提升授权率）
- **强制更新 vs 推荐更新**：重大版本强制（不可跳过），常规版本推荐（可跳过）
- **V1 不合并通知**：每次互动单独推送，V2 考虑批量合并

#### 如需修改通知时的检查清单

- [ ] 新增通知场景（如疫苗提醒 V2）→ 新增 FR-22X 并补充本表
- [ ] 修改权限申请时机 → 修改 FR-22D 触发节点

- [ ] 修改强制更新判断逻辑 → 修改 FR-22C 版本控制规则

### ⚙️ 系统基础功能（权限 / 举报 / 兽医离线 / 用户主页）
> 对应 PRD §4.7 FR-22D、§4.9 FR-25、§4.10 FR-26，以及 FR-4B 离线态更新。

#### 决策汇总

| 问题 | V1 决策 | V2 规划 |
|------|--------|---------|
| 相机/相册权限拒绝后 | **弹窗引导去系统设置手动开启** | — |
| 推送权限申请时机 | **第一次问诊完成后**（非冷启动）| — |
| 兽医离线状态显示 | **概率性**（时段可用性，非实时人数）| — |
| 兽医无人在线时 | **软引导回 AI 问诊**，不强制跳转 | 预约提醒功能 |
| 内容举报范围 | **仅内容举报**（不含评论举报、用户举报）| 评论举报、用户举报 |
| 举报处理 | **进运营人工审核队列** | 关键词自动过滤 |
| 点击他人头像 | **迷你预览卡**（不跳主页）文案「筹备中」| 完整用户公开主页 |

#### 「迷你主页预览卡」文案原则

> **不可以用**：「功能开发中」「敬请期待」「暂不支持」「该功能尚未开放」
>
> **应当用**：「TA 的专属主页正在精心筹备中，更多精彩即将呈现 ✨」

**核心逻辑**：让用户感受到产品在用心打造，而非感受到功能缺失。

### 🏥 专业问诊（Konsultasi Kilat）

| # | 功能 | 核心说明 |
|---|------|---------|
| 1 | **AI 多模态分诊** | 用户上传宠物照片/视频，AI 给出危险等级评估（绿/黄/红） |
| 2 | **黄色区间条件倒计时** | AI 不说「不严重」，只说「接下来X小时是重点观察期」，给出明确升级触发条件（如：「2小时内若再次出现→立即就医，若未出现→明早预约即可」） |
| 3 | **升级问诊入口** | AI 评估后弹出：「不放心？想确定用药方案？找权威医生还能帮你做什么」 |
| 4 | **兽医在线图文问诊** | Rp30,000/次；AI 问诊费可抵扣（已付 18K 的用户升级只需补差价 12K，非重新付 30K） |
| 5 | **新手兽医支撑系统** | 接单时可查看同类症状的历史判断数据库；无历史数据时提供 AI 参考答案；对话框显示免责提示「已告知用户本建议为参考，最终决策权在用户」 |
| 6 | **实时收入显示** | 兽医接单时对话框显示本次纯收入（如：「本单收入：Rp18,000」） |
| 7 | **V1 供给策略（运营）** | 平台 40% / 兽医 60%，吸引 IPB/UGM 网红兽医学生作为 KOL 自发传播 |

**定价结构：**
- AI 问诊：Rp18,000/次
- 专业兽医问诊：Rp30,000/次
- AI 问诊升级兽医：补差价 Rp12,000（不是重新付 30K）

**核心设计原则：**
- 平台是「工具提供方」，不是「医疗建议方」，责任在用户和兽医之间
- 用户自主决策，平台提供双向反馈渠道（举报不专业兽医 / 兽医申诉恶意差评）

- 数据沉淀 = 用户转换成本（宠物的历史记录在这里，不愿意换 App）

## 三、关键洞察与设计决策

### 专业问诊核心洞察

1. **「黄色区间」是最难设计的区间**——用户不需要「不严重可以等」，需要「接下来怎么观察、什么情况立刻行动」的可执行协议

2. **两类付费用户截然不同**
   - 焦虑型：AI 说可以等，但需要真人确认才能安心（买的是情绪）
   - 务实型：需要具体用药方案（买的是可执行建议）

3. **二次收费触发「敲诈感」**——解法：AI 问诊费抵扣，升级只补差价

4. **兽医供给经济模型已验证**：碎片时间兼职月收入 Rp3.7M > 全职助理月薪 Rp2-3M；V1 用 60% 分成 + 实习证书吸引 KOL 兽医学生

5. **平台定位清晰**：工具方 ≠ 医疗责任方；AI + 兽医签名维持专业感知层

### 宠物活动攒局核心洞察

6. **现有活动已是商业驱动**——雅加达宠物活动由场地+品牌联合举办，Instagram 发帖，WhatsApp 报名；平台角色是「聚合入口」而非「创造市场」

7. **三边市场**：宠物主（参与）× 组织者（工具需求）× 品牌/场地（曝光需求）

8. **V1 先给工具不推变现**——免费群聊+报名管理先让 Kevin 依赖平台，V2 再商业化

### 成长档案核心洞察

9. **攒局是成长档案的内容生产机器**——群聊照片认领机制让活动自动生成档案内容

10. **对外宠物名片是最强自然拉新工具**——「炫娃」动机 + 预览驱动下载，每个炫耀型用户都是分发渠道

11. **数据壁垒 > 功能壁垒**——宠物的历史记录积累越多，用户迁移成本越高；V2 的问诊历史+健康趋势会进一步锁定

## 四、三个功能的连接点（重要）

**V1 两功能连接点：**

```
专业问诊
    │ 问诊结束 → 用户确认 → 存入健康档案
    ↓
成长档案
    │ 宠物名片 → 分享 → 拉新
    └ 月费会员打通两个功能（V2）
```

**V2 加入攒局后的三功能闭环：**

```
专业问诊 → 成长档案 ← 群聊照片认领 ← 宠物活动攒局
```

V1 通过「问诊记录存档」形成数据闭环；V2 攒局加入后，「群聊照片认领」构成内容生产机器，月费会员打通三个功能实现商业闭环。
