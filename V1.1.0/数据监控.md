## 四、监控SQL查询（PawCoin 浮存余额 / 红色评级超额使用）

**方言**：PostgreSQL（未提供实际schema，按 AB-6C / FR-43C / AB-7A 字段含义假设了以下表结构，落地时请对照真实建表语句调整字段名）

**假设表结构**：
- `paw_coin_accounts(user_id, balance, updated_at)` — 用户 PawCoin 当前余额（IDR计价）
- `paw_coin_transactions(txn_id, user_id, type, amount, created_at)` — type ∈ recharge / consume / refund / withdraw
- `ai_consultations(consultation_id, user_id, pet_id, rating, quota_status, created_at)` — rating ∈ red/yellow/green；quota_status ∈ free / paid / overage_blocked（已超出当月免费额度后的状态）

### 1. PawCoin 浮存余额监控（对应 AB-6C）

```sql
-- 平台当前 PawCoin 浮存总额，对照印尼电子货币豁免门槛（约10亿IDR）计算占比，
-- 70%/90% 两档预警，超过后需提示运营评估是否申请支付牌照或限制充值
WITH float_balance AS (
    SELECT SUM(balance) AS total_float_idr
    FROM paw_coin_accounts
    WHERE balance > 0
)
SELECT
    total_float_idr,
    1000000000 AS regulatory_threshold_idr,                 -- 约10亿IDR豁免门槛
    ROUND(total_float_idr / 1000000000.0 * 100, 2) AS pct_of_threshold,
    CASE
        WHEN total_float_idr >= 900000000 THEN 'ALERT_90'   -- 90%红色预警
        WHEN total_float_idr >= 700000000 THEN 'WARN_70'    -- 70%黄色预警
        ELSE 'OK'
    END AS alert_level
FROM float_balance;
```

```sql
-- 近30日 PawCoin 每日充值/消耗/退款明细，对应后台 AB-6C「近30天每日浮存余额变化折线图」设计
SELECT
    DATE(created_at) AS txn_date,
    SUM(amount) FILTER (WHERE type = 'recharge')            AS recharge_idr,
    SUM(amount) FILTER (WHERE type IN ('consume', 'withdraw')) AS outflow_idr,
    SUM(amount) FILTER (WHERE type = 'refund')               AS refund_idr,
    SUM(amount) FILTER (WHERE type = 'recharge')
        - SUM(amount) FILTER (WHERE type IN ('consume', 'withdraw', 'refund')) AS net_float_change_idr
FROM paw_coin_transactions
WHERE created_at >= CURRENT_DATE - INTERVAL '30 days'
GROUP BY 1
ORDER BY 1;
```

**假设说明**：
- `balance` 为当前可用余额（已扣除消耗/退款），浮存总额=所有用户余额之和，即平台对用户的资金负债
- 阈值写死为常量，落地时建议放到配置表，避免硬编码
- 按 AB-6C 原始设计，后台视图需同时展示「枚数」与「等值IDR」两种单位，本查询只取IDR，落地时建议两列都查出来供前端展示
- AB-6C 已明确浮存余额为 **T+0 实时刷新**，预警渠道为「后台通知 + 邮件」，70%为黄色提醒、90%为红色高优先级告警——这部分口径已确定，不是开放问题

### 2. 红色评级超额使用监控（对应 FR-43C / AB-7A）

> AB-7A 后台视图明确要求三个时间维度的统计口径：**近7天触发次数（额度内/超额分列）**、**本自然月超额次数**、**账号生命周期累计超额次数**。PRD **未定义具体告警阈值**（FR-43C/OQ-12：处罚机制待上线后结合真实数据制定），下方查询只做数据采集和呈现，不内置阈值判断。

```sql
-- 对应 AB-7A 列表视图：每个曾触发超额使用的用户，按三个时间维度统计红色评级次数
WITH red_consultations AS (
    SELECT
        user_id,
        created_at,
        quota_status
    FROM ai_consultations
    WHERE rating = 'red'
)
SELECT
    user_id,
    COUNT(*) FILTER (
        WHERE created_at >= CURRENT_DATE - INTERVAL '7 days' AND quota_status = 'free'
    ) AS red_count_7d_within_quota,
    COUNT(*) FILTER (
        WHERE created_at >= CURRENT_DATE - INTERVAL '7 days' AND quota_status IN ('paid', 'overage_blocked')
    ) AS red_count_7d_overage,
    COUNT(*) FILTER (
        WHERE quota_status IN ('paid', 'overage_blocked')
        AND created_at >= DATE_TRUNC('month', CURRENT_DATE)
    ) AS overage_count_this_month,
    COUNT(*) FILTER (
        WHERE quota_status IN ('paid', 'overage_blocked')
    ) AS overage_count_lifetime,
    MAX(created_at) AS last_triggered_at
FROM red_consultations
GROUP BY user_id
HAVING COUNT(*) FILTER (WHERE quota_status IN ('paid', 'overage_blocked')) > 0
ORDER BY red_count_7d_overage DESC;
```

```sql
-- 单账号详情（对应 AB-7A「点击进入触发历史记录列表」）
SELECT
    consultation_id,
    created_at AS triggered_at,
    rating,
    quota_status
FROM ai_consultations
WHERE user_id = :user_id
    AND rating = 'red'
ORDER BY created_at DESC;
```

```sql
-- 排查用：短时间窗口内高频触发，用于辅助判断是否需要人工标记「待核查」
-- 阈值（1小时内≥3次）非PRD定义，是按 v1-1-0PRD.md §3.1 末尾"MVP阶段建议补充粗粒度限速"
-- 段落里给出的"1小时内"窗口示例改写的排查脚本，仅供运营参考，不构成自动拦截逻辑
SELECT
    user_id,
    COUNT(*) AS red_count_in_window,
    MIN(created_at) AS window_start,
    MAX(created_at) AS window_end
FROM ai_consultations
WHERE rating = 'red'
    AND quota_status IN ('paid', 'overage_blocked')
    AND created_at >= NOW() - INTERVAL '1 hour'
GROUP BY user_id
HAVING COUNT(*) >= 3
ORDER BY red_count_in_window DESC;
```

**假设说明**：
- `quota_status` 需要在问诊记录写入时打标，区分"免费额度内/已超额付费/已超额被拦截"三种状态，目前PRD未明确字段名，落地前需与后端确认
- AB-7A 视图还要求「账号状态」字段（正常/已标记待核查/已处理），本查询未建模该状态表，落地时需要补一张 `account_review_flags(user_id, status, note, updated_at)` 之类的标记表
- 第三条查询的阈值（1小时内≥3次）是排查脚本示例，不是PRD规定的处罚触发线，正式阈值待OQ-12在上线后结合真实数据确定

## 五、数据刷新频率口径

复核 v1-1-0后台prd.md 时发现两处 `[NOTE FOR PM]`，原文未定数据是 T+0 准实时还是 T+1 批量（AB-7A 红色评级超额使用监控、AB-8C AI问诊收入统计）。**本文档统一按 T+1（次日批处理）口径定义**，即：

- 「日监控」表中「红色评级超额使用次数」「当日GMV」「AI问诊付费转化率」等依赖 AB-7A / AB-8C 的指标，实际看到的是**前一天完整数据**，运营查看时按"昨日数据"理解，不强求当日实时
- 上方 SQL 查询中 `CURRENT_DATE - INTERVAL '1 day'` 的取数窗口已经是按"昨日一整天"设计的，与T+1口径一致，无需改动
- 若后续工程评估T+0准实时成本可接受，可以再升级，但作为V1.1.0上线门槛不强制
- PawCoin 浮存余额监控（AB-6C）例外：PRD已明确为T+0实时刷新，不受本条T+1默认口径影响
