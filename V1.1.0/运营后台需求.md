## 1. 背景与目标

V1.0.0 管理后台上线并投入运营试用后，暴露出三类需要解决的问题：

1. **可用性问题**：举报队列、内容管理、用户管理、兽医账号等模块在实际操作中存在效率瓶颈（如缺少重置筛选、缺少批量操作确认、列表信息展示不清晰等），运营团队在日常使用中已明确指出具体整改点。
2. **种子内容发布效率**：平台上线前及日常运营需要持续发布种子内容以维持 Feed 活跃度，但当前"单条填写图片 URL"的发布方式效率低，无法支撑批量铺量。
3. **问诊服务开始收费**：自 V1.1.0 起，兽医问诊与 AI 问诊服务开始收费 `[ASSUMPTION: 具体计费规则、支付渠道接入由其他技术方案覆盖，本文档仅覆盖后台对"问诊订单"数据的展示、查询与导出需求，不涉及支付网关对接的技术实现]`。运营需要在后台看到订单量、成交额等财务相关数据，并能按订单排查问题。

V1.1.0 的目标是：在不改变 V1.0.0 已定的信息架构原则（登录体系、审计留痕、扁平/两级权限模型）的前提下，完成上述三类整改，使后台可支撑运营团队更高效地完成日常工作，并为问诊收费上线做好数据可见性准备。

### 4.2 模块 3（扩展）：种子内容发布

#### AB-1.1-02：批量上传功能（帖子+图文）

**需求描述：** 支持运营通过"zip 图片包 + Excel"的组合方式批量发布种子内容，替代逐条手动发布。

**运营操作流程：**

1. **整理图片文件夹**：本地按规则组织图片并打包为 zip：
   ```
   待上传图片/
     ├── sku001/          ← 文件夹名 = 产品编码（不用中文/特殊字符）
     │     ├── 1.jpg      ← 数字前缀表示图片顺序
     │     ├── 2.jpg
     │     └── 3.jpg
     ├── sku002/
     │     ├── 1.jpg
     │     └── 2.jpg
   ```
2. **生成骨架 Excel**：后台上传 zip，点击【生成模板】，系统扫描文件夹并生成 Excel——"产品编码"列自动填文件夹名，"图片"列自动填该文件夹内文件名列表（如 `1.jpg,2.jpg,3.jpg`）；系统同时记录该 zip 的文件清单指纹（如文件夹+文件名列表的哈希），用于步骤 4 校验。
3. **补充内容并提交**：运营下载骨架 Excel，补充标题、正文、**内容类型**（`DAILY` 或 `KNOWLEDGE`，按行填写）等文字字段（图片列不改），完成后**同时上传**补充好的 Excel + 步骤 1 的 zip 包（可以是同一个 zip，也可以是内容完全一致的重新打包）。
4. **校验预览后确认入库**：系统先展示校验结果（匹配成功/失败行数及原因），运营确认无误后点击【确认导入】才正式入库。

**功能需求清单：**

| 编号 | 功能点 | 说明 |
|---|---|---|
| F1 | 骨架 Excel 生成 | 上传 zip，扫描子文件夹，自动生成预填"产品编码+图片文件名列表"的 Excel |
| F2 | Excel+zip 联合提交入口 | 支持同时上传 Excel 和 zip |
| F3 | 图片匹配引擎 | 按 Excel 每行的产品编码+文件名，在 zip 内定位对应图片；同时比对该 zip 的文件清单指纹与生成骨架 Excel 时记录的指纹是否一致，防止提交时误传了文件名相同但内容不同的 zip |
| F4 | 服务器唯一文件名生成 | 匹配成功的图片生成唯一 Key 后上传，避免同名覆盖 |
| F5 | URL 回填 | 上传成功后，按原顺序把真实 URL 替换进 Excel 图片列 |
| F6 | 校验预览 | 展示匹配成功/失败行数，失败行标注具体缺失文件名 |
| F7 | 确认入库 | 校验通过后需运营手动确认才写入正式数据 |

**服务器存储命名规则（避免重名覆盖）：** 原始文件名（如 `1.jpg`）仅用于匹配，不能直接作为服务器存储文件名：

```
{产品编码}_{批次号/时间戳}_{序号}_{短哈希}.jpg
示例：sku001_20260701_1_a3f9c2.jpg
```
- 产品编码：便于按帖子追溯
- 批次号/时间戳：避免同一产品重复上传时覆盖旧图
- 序号：保留原始顺序
- 短哈希（基于文件内容）：最终防冲突保险

**异常场景处理：**

| 场景 | 处理方式 |
|---|---|
| Excel 图片列文件名在 zip 中找不到 | 校验预览标记失败，注明缺失文件名，不允许入库该行 |
| 文件夹名重复 | 生成骨架 Excel 时报错，要求修正后重新打包 |
| 图片格式/大小不符 | 匹配阶段校验，超限标记失败原因 |
| 重复提交同批次 | 批次号+哈希机制保证不覆盖历史图片 |
| 提交的 zip 与生成骨架 Excel 时的 zip 文件清单指纹不一致（如误传了另一个 zip） | 校验预览标记整批失败，提示"zip 内容与生成模板时不一致，请重新上传"，不允许入库 |

**内容类型范围：** 批量上传覆盖 `DAILY`、`KNOWLEDGE` 两种内容类型；`GROWTH_MOMENT`（成长日历快乐时刻，需绑定宠物 ID 且须属运营账号）不纳入批量上传范围，继续通过单条发布（AB-1.1-03）完成。

**验收标准：**
- [ ] 上传 zip 可一键生成图片列已预填的 Excel 模板
- [ ] Excel+zip 联合提交可自动完成图片上传与 URL 回填
- [ ] 校验预览能准确识别图片匹配失败的行及原因
- [ ] 服务器存储文件名全局唯一，不因文件夹内同名文件（如 1.jpg）产生覆盖
- [ ] 运营全程无需手动复制/拼接图片 URL

**上传限制：** zip 包大小上限 300 MB；文件夹（产品编码）数量不限制。超出大小上限的上传，需在校验阶段拦截并提示。

**需求描述：** 优化线上已存在的"发布种子内容"单条发布表单（该表单未在 V1.0.0 PRD 中正式记录，见 §0 说明；不要与 AB-3D「种子内容账号打标签」能力混淆）。

## 5. 非功能需求

- **订单 ID 隔离**：兽医问诊订单与 AI 问诊订单的订单 ID 命名空间必须相互独立，系统层面禁止重复（AB-1.1-27）
- **操作留痕**：批量导入（AB-1.1-02）、批量下架/驳回（AB-1.1-05）、内容下架（AB-1.1-11）等操作需计入操作审计日志，延续 V1.0.0 AB-1B 的审计要求
- **数据一致性**：批量上传的图片匹配、URL 回填过程需保证失败行不会造成部分入库（要么整行成功，要么标记失败不入库）
- **导出方式**：内容管理、虚拟用户、问诊订单等导出功能在 V1.1.0 采用同步直接下载（点击导出后浏览器直接生成并下载文件），暂不做异步生成+下载通知机制；当前数据量较小，待未来数据量增长导致导出耗时/超时问题后再升级为异步方案

## 7. 成功指标

| 指标 | 说明 |
|---|---|
| 种子内容发布人力投入 | 批量上传上线后，运营发布同等数量种子内容所需操作时间应显著下降 |
| 举报/内容审核处理效率 | 状态列、批量确认、查看详情等整改上线后，运营处理一条举报/内容所需点击次数与耗时下降 |
| 问诊订单财务数据可得性 | 运营可直接通过后台获取准确的问诊 GTV/成交额数据，无需人工对账 |

**反向指标（Counter-metrics）：**
- 批量上传上线后，图片误覆盖/错配导致的数据纠错工单数应为 0（对应验收标准）
- 导出数据需与列表页展示状态一致，不因异步/缓存产生偏差

`[ASSUMPTION]` 上表指标方向明确但未给出量化目标（如"下降 XX%"）——当前无历史操作耗时基线数据支撑设定具体数字，建议 V1.1.0 上线后先采集一个统计周期的实际数据，再回填量化目标。
