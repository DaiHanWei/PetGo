2. **问诊订单模型**：运营文档提出的"兽医问诊订单"" AI 问诊订单"两个查询入口，与本文档模块 8（收费运营数据）的 AB-8B/AB-8C/AB-8E 是同一件事的不同粒度描述，已合并统一，不重复建模（见 §4 模块 8）。
3. **订单状态词表**：运营文档的状态词（待接单/进行中/关闭/异常）基于对收费逻辑的旧理解；已按 FR-43D 2026-07-06 修订版（免费入队、接单后限时支付，不存在"已暂停"中间态）重新定义 AB-8B 状态机（见 §4 AB-8B）。
4. 运营文档中"模块 8：概览看板""模块 9（新增）：问诊订单管理"两个模块编号与本文档已有的模块 8（收费运营数据）、模块 9（用户标识字段异步审核）冲突，已重新编号（概览看板 → 模块 10；问诊订单管理内容并入模块 8，不单独设模块）。
5. **虚拟账号机制**：运营提出的"发布账号选择器/虚拟用户列表"需要一个可枚举、可挑选多个身份的账号池，但经查代码库确认现状是——种子内容一律以当前登录运营自己的身份发布，且 2026-06-29 已有明确决策不引入账号打标签机制（原 AB-3D）。本次经确认**推翻该决策**，新增 AB-1.1-02A 虚拟账号管理机制（见 §9.1），这是一项新增工程量，不是简单的整改。

### AB-5A：反馈工单列表

**后果：**

- 工单列表字段：工单 ID、用户账号、提交时间、分类标签、状态、负责客服、关联订单（若有）

- 支持按用户 ID 或工单内容关键词搜索

---

### AB-5B：工单详情与客服处理

**后果：**

**详情视图展示：**

- 用户提交的完整问题描述
- 图片附件（用户上传，支持在后台内预览）
- 用户填写的联系方式（邮箱或 WhatsApp 电话，醒目展示，客服主要联系渠道）
- 内部备注记录（客服历史操作备注，用户不可见）
- **关联订单**（选填）：客服可搜索并关联该工单对应的 AB-8B 兽医咨询订单（按订单 ID 或用户账号搜索）；分类标签（AB-5C）选中"退款申请"时，界面提示客服关联订单

**客服操作：**

- **内部备注**：客服可添加处理备注（用户不可见），按时间正序排列
- **「是否需要联系客户」开关**：客服判断此工单是否需要电话 / WA / 邮件跟进，默认关闭
- **「已联系客户」勾选**：客服完成站外联系后勾选；与「标记已解决」联动触发 CSAT（见 AB-5H）
- **「标记已解决」按钮**：触发用户侧结案通知（FR-52A）；若「已联系客户」已勾选，同时触发 CSAT 调查（FR-52B）
- **「批准退款需求」按钮**（对应 FR-43D 客服判定环节，需先完成"关联订单"）：点击需二次确认（弹窗展示关联订单的金额等信息）；确认后：① 关联订单状态置为「退款处理中」（AB-8B）；② App 端该订单详情页随之解锁「选择退款方式」入口（FR-43D）；**不发送系统通知**——用户下次查看该订单时会自然看到「选择退款方式」入口，无需额外推送；操作记录操作人和时间戳，可填写内部备注（选填）。此按钮只代表"同意退款需求"，不等同于工单"已解决"——工单需等用户完成「选择退款方式」及后续 AB-5E 审批/打款流程后再标记已解决
- **「驳回退款需求」按钮**：点击须填写驳回原因（必填，内部可见，不对用户展示具体原因）；**驳回后系统向用户发送通知，告知退款需求未通过**（App 端没有其他 UI 变化可以让用户感知结果，必须主动告知）——标题「退款申请未通过」，正文「很抱歉，你的退款申请未通过审核，如有疑问可通过「联系客服」进一步沟通」，跳转对应工单详情（见 App PRD 通知文案汇总表）；驳回后工单保持「处理中」，客服可继续跟进或改用其他方式结案

### AB-5D：工单状态管理

**后果：**

### AB-5E：退款申请流程

*依赖支付基础设施（V1.1.0 建设内容）；付费问诊上线后启用。*

**两种创建方式，共用同一套审批流转：** 对应用户端 PRD FR-43D——用户联系客服（FR-52 工单）说明退款需求，客服判定同意后，App 订单详情页才出现「选择退款方式」入口；用户在该入口自主选择退款方式并填写账户信息提交后，**系统直接生成退款申请，不需要客服再手动创建一遍**。客服手动创建仅用于工单场景之外的例外情况（如用户无法自助操作、需要客服代为处理）。

```text
方式 A · 客服手动创建（例外场景）
客服在工单详情页填写退款申请（金额 + 原因 + 关联工单 + 用户账号）
  ↘
方式 B · 系统自动创建（常规场景，对应 FR-43D）
用户在 App 订单详情页「选择退款方式」提交（退款渠道 + 账户信息）
  ↘（原因/关联工单自动带入客服已批准的 FR-52 工单）
    → 主管收到待审批通知 → 审批通过 / 驳回
      → 财务看到已审批列表 → 标记已打款
        → 客服可查看全程状态
```

**后果：**

- **方式 A（客服手动创建）**：被授权提交退款申请的客服填写：退款金额、货币（IDR）、原因、关联工单 ID、用户账号
- **方式 B（系统自动创建，常规路径）**：用户完成 App 端「选择退款方式」提交后，系统自动生成退款申请，自动带入：关联订单 ID、退款金额（按订单实付金额）、货币、用户账号、原因（引用原始 FR-52 工单内容）；**无需客服创建这一步**，申请直接进入「待主管审批」状态
- **用户填写的退款账户信息在详情页显眼展示**（因印尼支付环境无法系统追溯原付款渠道，财务须凭此信息手动打款）：退款渠道 / 账号或手机号 / 账户持有人姓名；方式 A 下若信息有疑问，客服通过用户填写的联系方式（FR-52 表单字段）确认后再提交审批
- 两种方式提交后均创建退款工单，记录来源（客服创建 / 用户自助提交）；被授权审批退款的主管收到后台内通知
- 主管审批通过或驳回，须填写处理备注（必填）
- 审批通过的工单进入财务队列；被授权标记打款的财务标记已打款并填写付款凭证
- 所有节点均记录操作人和时间戳；方式 A 下客服随时可查看全程流转状态；方式 B 下用户可在 App 订单详情页查看状态（对应 FR-43D「退款处理中」/「已退款」展示）
- 驳回的申请：方式 A 含主管备注返回给客服，客服可修改后重新提交；方式 B 含主管备注通知用户，`[NOTE FOR PM：用户自助提交被驳回后能否在 App 内重新提交，还是必须转回联系客服处理，需在 UX 阶段确定]`

## 2. 模块 6：PawCoin 管理（V1.1.0 新增）

对应用户端 PRD：V1.1.0 FR-50 系列（PawCoin 体系）、FR-43D（退款选择界面）

**功能描述：** V1.1.0 引入 PawCoin 虚拟货币体系（法务已审查通过，见 OQ-15 ✅）。管理后台配套建立 PawCoin 运营配置能力：退款溢价比例动态配置（AB-6A）、充值档位管理（AB-6B）、浮存余额监控（AB-6C）。

---

### AB-6A：PawCoin 退款溢价比例配置

对应用户端 PRD：FR-43D 退款选择界面——选项 A「转为 PawCoin 余额」中，溢价补偿枚数的计算依据。

**后果：**

- 配置入口：管理后台「PawCoin 管理」→「退款溢价设置」
- 配置字段：**溢价比例**（整数，单位 %，例：填写 `10` 表示应退金额额外多给 10% 的 PawCoin 枚数）
  - 允许范围：0%～50%；不允许负值；默认值由运营团队在上线前录入 `[NOTE FOR PM: 初始溢价值待运营团队确认，建议参考行业惯例（如 5%～15%）并结合现金流目标设定]`
- **生效时机**：修改并保存后立即对新触发的退款申请生效；已进入「申请退款」选择界面但尚未提交选择的存量会话，展示的溢价枚数按**会话打开时**读取到的比例计算，不会在用户操作过程中动态刷新（避免用户看到枚数变化产生困惑）
- **历史比例快照**：每笔实际通过选项 A 完成的退款，系统记录该笔交易使用的溢价比例，供对账核查；配置修改不影响历史已完成的退款记录
- **操作权限**：仅限「运营管理员」角色，普通客服账号不可见此配置页
- **变更日志**：每次修改自动记录操作人、操作时间、变更前后值，保留完整历史，供运营和财务审计查阅

---

### AB-6B：充值档位配置

对应用户端 PRD：FR-50A 充值获取（购买）——用户侧仅展示运营预设档位，不支持自定义金额。

**后果：**

- 配置入口：管理后台「PawCoin 管理」→「充值档位设置」
- **档位列表**：展示当前全部档位，每条档位字段：充值金额（IDR）/ 对应 PawCoin 枚数（系统自动按 1:1 计算，不可手动填写）/ 启用状态 / 显示顺序
- **新增档位**：填写充值金额（IDR）→ 枚数自动计算（显示验证：N IDR = N 枚）→ 设置显示顺序 → 保存后对用户侧即时生效
- **启用 / 禁用**：运营可随时切换任意档位的启用状态；**规则：充值档位页至少须保留 1 个启用状态的档位**，系统不允许将全部档位同时禁用（操作时弹出阻断提示）
- **显示顺序**：支持拖拽排序；用户侧充值档位选择页按此顺序展示
- **操作权限**：仅限运营管理员
- **变更日志**：记录每次新增 / 禁用 / 顺序调整的操作人和时间，保留历史供审计查阅
- **上线初始档位已确定为 4 档：Rp10,000 / Rp25,000 / Rp50,000 / Rp100,000**（对应用户端 PRD A-14），运营团队须在上线前通过本配置页录入这 4 档并验证用户侧展示正常；上线后档位本身可随时通过本页增删调整，无需改代码

---

### AB-6C：PawCoin 浮存余额监控

对应用户端 PRD：FR-50A——平台须持续监控全平台 PawCoin 未消费余额总量（浮存），避免触及印尼闭环电子货币豁免门槛（约 Rp10 亿）。

**后果：**

- 入口：管理后台「PawCoin 管理」→「浮存余额监控」
- **实时概览**：
  - 全平台当前 PawCoin 未消费余额总量（单位：枚，等值 IDR，T+0 刷新）
  - 与法规豁免门槛（Rp10 亿）的距离，以进度条 + 百分比展示（如「已使用 23.4%，剩余 Rp7.66 亿空间」）
  - 本自然月净新增浮存（本月充值总量 - 本月消费总量）
- **趋势图**：近 30 天每日浮存余额变化折线图，辅助判断增长趋势
- **预警机制**：
  - 浮存超过门槛 **70%**：系统向运营管理员发送后台通知 + 邮件告警（黄色，提示"建议提前评估是否需要启动牌照申请"）
  - 浮存超过门槛 **90%**：红色高优先级告警（同渠道），提示"须立即评估是否暂停充值入口或启动正式牌照申请"
- **操作说明**：本视图仅供监控和预警，暂停充值入口须工程手动操作（V1.1.0 不提供后台一键暂停充值能力）；启动牌照申请须线下走合规流程，后台不直接支持

## 3. 模块 7：AI 问诊运营数据（V1.1.0 新增）

对应用户端 PRD：V1.1.0 FR-43C（红色评级安全护栏，超额使用监测）

**功能描述：** 红色评级安全提醒在任何情况下均免费展示（FR-43C 不可妥协护栏），但额度用尽后触发的红色评级记录为「超额使用」，供运营后台查看和分析，用于识别异常高频触发账号，辅助后续支付策略调整决策。

---

### AB-7A：红色评级超额使用监控视图

**后果：**

- 入口：管理后台「AI 问诊数据」→「红色评级超额使用」
- **列表视图**：展示触发过超额使用的用户列表，字段包括：
  - 用户账号 / 注册时间
  - 最近 7 天红色评级触发次数（含额度内 + 超额，分列展示）
  - 超额使用次数（本自然月）
  - 累计超额使用次数（账号生命周期）
  - 最近一次触发时间
  - 当前账号状态（正常 / 已标记待核查 / 已处理）
- **筛选与排序**：支持按时间范围、超额次数区间、账号状态筛选；默认按最近 7 天红色触发次数降序排列

- **筛选**：支持按时间范围、订单状态、支付方式、待核查标记筛选；支持按用户账号、订单 ID 或**接诊兽医账号**搜索
- **订单详情**：点击展开完整记录，含列表字段全部信息 + 各阶段时间戳（每次接单时间 / 每次支付尝试的成功或失败时间 / 会话开始时间 / 会话结束时间 / 退款申请提交时间 / 退款处理完成时间，按订单实际经过的阶段展示，未发生的阶段留空；若订单经历多次"接单→未支付→回待接单"循环，各次接单/失败时间戳均需保留，不覆盖历史记录）

- **不提供退款操作入口**：退款处理须通过客服 AB-5E 工单流程发起，本视图仅展示状态

## 9. 模块 3（扩展）：内容与用户运营整改（运营需求合并新增）

对应原 AB-3A（举报队列）、AB-3B（内容管理）、AB-UA-01（用户管理），以及种子内容发布能力，运营试用后台后提出的整改点。

### 9.1 种子内容发布

#### AB-1.1-02：批量上传功能（帖子+图文）

支持运营通过"zip 图片包 + Excel"的组合方式批量发布种子内容，替代逐条手动发布。

**运营操作流程：**
1. **整理图片文件夹**：本地按规则组织图片并打包为 zip，文件夹名 = 产品编码（不用中文/特殊字符），数字前缀表示图片顺序
2. **生成骨架 Excel**：后台上传 zip，点击【生成模板】，系统扫描文件夹并生成 Excel——"产品编码"列自动填文件夹名，"图片"列自动填该文件夹内文件名列表；系统同时记录该 zip 的文件清单指纹，用于步骤 4 校验
3. **补充内容并提交**：运营下载骨架 Excel，补充标题、正文、内容类型（`MOMENT` 或 `KNOWLEDGE`）、**发布账号**（填写 AB-1.1-02A 虚拟账号的昵称或账号 ID，逐行指定，同一批次内不同行可以是不同虚拟账号）等文字字段，完成后同时上传补充好的 Excel + zip 包
4. **校验预览后确认入库**：系统先展示校验结果（匹配成功/失败行数及原因），运营确认无误后点击【确认导入】才正式入库

**功能需求清单：**

| 编号 | 功能点 | 说明 |
|---|---|---|
| F1 | 骨架 Excel 生成 | 上传 zip，扫描子文件夹，自动生成预填"产品编码+图片文件名列表"的 Excel，并预留"发布账号"空列供运营逐行填写 |
| F2 | Excel+zip 联合提交入口 | 支持同时上传 Excel 和 zip |
| F3 | 图片匹配引擎 | 按 Excel 每行的产品编码+文件名，在 zip 内定位对应图片；同时比对 zip 文件清单指纹与生成骨架 Excel 时的指纹是否一致 |
| F3A | 发布账号校验 | 按 Excel 每行的"发布账号"字段匹配 AB-1.1-02A 虚拟账号（按昵称或账号 ID），校验账号存在且处于启用状态 |
| F4 | 服务器唯一文件名生成 | 匹配成功的图片生成唯一 Key 后上传，避免同名覆盖 |
| F5 | URL 回填 | 上传成功后，按原顺序把真实 URL 替换进 Excel 图片列 |
| F6 | 校验预览 | 展示匹配成功/失败行数，失败行标注具体缺失文件名或发布账号错误原因 |
| F7 | 确认入库 | 校验通过后需运营手动确认才写入正式数据 |

**服务器存储命名规则：** `{产品编码}_{批次号/时间戳}_{序号}_{短哈希}.jpg`（示例：`sku001_20260701_1_a3f9c2.jpg`），避免同名覆盖。

**异常场景处理：**

| 场景 | 处理方式 |
|---|---|
| Excel 图片列文件名在 zip 中找不到 | 校验预览标记失败，注明缺失文件名，不允许入库该行 |
| 文件夹名重复 | 生成骨架 Excel 时报错，要求修正后重新打包 |
| 图片格式/大小不符 | 匹配阶段校验，超限标记失败原因 |
| 重复提交同批次 | 批次号+哈希机制保证不覆盖历史图片 |
| 提交的 zip 与生成骨架 Excel 时的 zip 文件清单指纹不一致 | 校验预览标记整批失败，不允许入库 |
| "发布账号"填写的昵称/ID 不存在，或对应虚拟账号已被禁用 | 校验预览标记失败，注明具体行号及原因，不允许入库该行 |
| "发布账号"列留空 | 校验预览标记失败，视为必填字段缺失，不允许入库该行 |

**内容类型范围：** 覆盖 `MOMENT`（原「日常分享」，App 端现已统一显示为「Moment」）、`KNOWLEDGE`；`GROWTH_MOMENT` 不纳入批量上传范围，继续通过单条发布（AB-1.1-03）完成。

`[NOTE FOR PM/工程：经查代码库，后端 ContentType 枚举目前实际值仍是 DAILY（未改名），App 侧只是把显示文案改成了「Moment」（ARB key 仍叫 publishSegmentDaily/feedTabDaily），枚举本身没有 MOMENT 这个值。本文档统一按 App 现有的用户可见名称写作 MOMENT，但骨架 Excel 里运营实际填写、系统实际入库时用的字符串，要按后端枚举现状对应到 DAILY，还是借这次批量上传顺带把枚举也改名为 MOMENT，需工程确认]`

**上传限制：** zip 包大小上限 300 MB；文件夹（产品编码）数量不限制，超出大小上限需在校验阶段拦截并提示。

#### AB-1.1-02A：虚拟账号管理（新机制，运营需求合并新增）

**背景与决策变更说明：** 经查代码库（`petgo-platform`），V1.0.0 规划阶段（决策 2026-06-29，见 `epics.md`）已明确否决"种子内容账号打标签"能力（原 AB-3D）：不引入 `users.is_seed_account` 列，不提供标记 UI；现状是种子内容一律以**当前登录运营自己**关联的 `role=ADMIN` 身份发布（按 Lark 邮箱 1:1 关联，若该运营账号未关联则直接报错，不会退而选用其他账号），不存在任何可枚举、可挑选的"虚拟账号列表"。运营在 AB-1.1-03 提出的"发布账号选择器：从虚拟账号列表中选择"，需要的正是一个可枚举多个独立身份的账号池——这与 2026-06-29 的决策在机制上冲突。**本次经确认，推翻该决策，正式引入独立的虚拟账号机制**，原因：批量发布种子内容需要让 Feed 呈现多个不同"作者"，而不是所有种子内容都挂在同一个运营登录身份下。

**功能描述：** 新增「虚拟账号」，作为独立于普通注册用户（`role=USER`）、后台运营账号（`role=ADMIN`）、兽医账号（`role=VET`）的第四类账号，专用于种子内容发布，不具备登录能力（无密码、无登录入口），仅作为内容作者身份存在。

**后果：**
- 入口：管理后台「用户管理」→「虚拟账号管理」（新增页面，与 AB-UA-01 真实用户列表是两个独立入口，不是同一列表的两个视图）
- **创建**：运营可随时新建虚拟账号，填写：昵称（必填，≤20 字）、头像（选填，默认使用系统预设占位图）
- **列表字段**：账号 ID / 昵称 / 头像 / 创建时间 / 创建人 / 启用状态 / 累计发布内容数
- **启用 / 禁用**：禁用后的虚拟账号不可再被选为新内容的发布者；其历史已发布内容的作者归属不变、内容本身不受影响
