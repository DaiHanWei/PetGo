## 1. 模块 5：客服与产品反馈（V1.1.0 新增）

对应用户端 PRD：V1.1.0 FR-52 系列（站内信投诉渠道）

**功能描述：** V1.1.0 上线管理后台客服工单处理能力。V1.0.0 用户侧的帮助反馈入口（FR-20）在此版本升级为站内信投诉渠道（FR-52），后台同步建立工单处理、结案通知触发和满意度调查管理机制。

---

### AB-5A：反馈工单列表

**后果：**

- 通过 App 帮助反馈入口提交的所有工单在此展示（对应 FR-52）
- 工单列表字段：工单 ID、用户账号、提交时间、分类标签、状态、负责客服
- 状态流转：**待处理** → **处理中** → **已解决** → **已关闭**
- 支持按状态、分类、日期范围、负责客服筛选
- 支持按用户 ID 或工单内容关键词搜索

---

### AB-5B：工单详情与客服处理

**后果：**

**详情视图展示：**

- 用户提交的完整问题描述
- 图片附件（用户上传，支持在后台内预览）
- 用户填写的联系方式（邮箱或 WhatsApp 电话，醒目展示，客服主要联系渠道）
- 内部备注记录（客服历史操作备注，用户不可见）

**客服操作：**

- **内部备注**：客服可添加处理备注（用户不可见），按时间正序排列
- **「是否需要联系客户」开关**：客服判断此工单是否需要电话 / WA / 邮件跟进，默认关闭
- **「已联系客户」勾选**：客服完成站外联系后勾选；与「标记已解决」联动触发 CSAT（见 AB-5H）
- **「标记已解决」按钮**：触发用户侧结案通知（FR-52A）；若「已联系客户」已勾选，同时触发 CSAT 调查（FR-52B）

---

### AB-5C：工单分类与标签

**后果：**

- 客服可为工单添加一个或多个分类标签：Bug 反馈 / 功能建议 / 问诊投诉 / 内容问题 / 账号问题 / 表扬 / 其他
- 标签供产品团队分析使用；V1.1.0 不做基于标签的自动化触发
- 标签分配后可随时修改

---

### AB-5D：工单状态管理

**后果：**

- 状态流转：待处理 → 处理中（客服手动切换）→ 已解决（客服标记）→ 已关闭（已解决后 7 天自动关闭，或客服手动关闭）
- 工单为单次提交，用户不可在原工单内追加回复；如需再次反馈，用户重新提交新工单
- SLA 指示器：「待处理」状态超过 24 小时的工单在列表中高亮显示（24 小时首次响应为内部运营目标）

---

### AB-5E：退款申请流程

*依赖支付基础设施（V1.1.0 建设内容）；付费问诊上线后启用。*

**三级审批流转：**

```text
客服提交退款申请（金额 + 原因 + 关联工单）
  → 主管收到待审批通知 → 审批通过 / 驳回
    → 财务看到已审批列表 → 标记已打款
      → 客服可查看全程状态
```

**后果：**

- 被授权提交退款申请的客服填写：退款金额、货币（IDR）、原因、关联工单 ID、用户账号
- **用户填写的退款账户信息在详情页显眼展示**（因印尼支付环境无法系统追溯原付款渠道，财务须凭此信息手动打款）：退款渠道 / 账号或手机号 / 账户持有人姓名；若信息有疑问，客服通过用户填写的联系方式（FR-52 表单字段）确认后再提交审批
- 提交后创建退款工单；被授权审批退款的主管收到后台内通知
- 主管审批通过或驳回，须填写处理备注（必填）
- 审批通过的工单进入财务队列；被授权标记打款的财务标记已打款并填写付款凭证
- 所有节点均记录操作人和时间戳；客服随时可查看全程流转状态
- 驳回的申请含主管备注返回给客服，客服可修改后重新提交

---

### AB-5F：产品反馈汇总视图

**后果：**

- 按分类标签汇总工单数量，支持日期范围筛选
- 各标签下工单数，支持按数量排序
- 设计目标：供产品团队人工查阅；V1.1.0 不做自动报表推送，后续版本按需考虑

---

### AB-5H：CSAT 满意度汇总视图

*触发条件：客服勾选「已联系客户」并标记「已解决」后，系统向用户发送满意度调查（FR-52B），用户提交后汇总至此视图。*

**后果：**

- 展示所有已收到满意度评分的工单汇总：平均分、评分分布（1-5 星各占比）、评分总数
- 支持按日期范围、工单分类标签、负责客服筛选
- 可查看每条评分详情：工单 ID、评分、文字评价（如有）、负责客服账号
- 设计目标：供客服主管和产品团队定期查阅，用于客服质量管理和流程优化；V1.1.0 不做自动报表推送

---

## 2. 模块 6：PawCoin 管理（V1.1.0 新增）

对应用户端 PRD：V1.1.0 FR-50 系列（PawCoin 体系）、FR-43D（退款选择界面）

**功能描述：** V1.1.0 引入 PawCoin 虚拟货币体系（法务已审查通过，见 OQ-15 ✅）。管理后台配套建立 PawCoin 运营配置能力：退款溢价比例动态配置（AB-6A）、充值档位管理（AB-6B）、浮存余额监控（AB-6C）。

---

### AB-6A：PawCoin 退款溢价比例配置

对应用户端 PRD：FR-43D 退款选择界面——选项 A「转为 PawCoin 余额」中，溢价补偿枚数的计算依据。

**后果：**

- 配置入口：管理后台「PawCoin 管理」→「退款溢价设置」
- 配置字段：**溢价比例**（整数，单位 %，例：填写 `10` 表示应退金额额外多给 10% 的 PawCoin 枚数）
  - 允许范围：0%～50%；不允许负值；默认值由运营团队在上线前录入 `[NOTE FOR PM: 初始溢价值待运营团队确认，建议参考行业惯例（如 5%～15%）并结合现金流目标设定]`
- **生效时机**：修改并保存后立即对新触发的退款申请生效；已进入「申请退款」选择界面但尚未提交选择的存量会话，展示的溢价枚数按**会话打开时**读取到的比例计算，不会在用户操作过程中动态刷新（避免用户看到枚数变化产生困惑）
- **历史比例快照**：每笔实际通过选项 A 完成的退款，系统记录该笔交易使用的溢价比例，供对账核查；配置修改不影响历史已完成的退款记录
- **操作权限**：仅限「运营管理员」角色，普通客服账号不可见此配置页
- **变更日志**：每次修改自动记录操作人、操作时间、变更前后值，保留完整历史，供运营和财务审计查阅

---

### AB-6B：充值档位配置

对应用户端 PRD：FR-50A 充值获取（购买）——用户侧仅展示运营预设档位，不支持自定义金额。

**后果：**

- 配置入口：管理后台「PawCoin 管理」→「充值档位设置」
- **档位列表**：展示当前全部档位，每条档位字段：充值金额（IDR）/ 对应 PawCoin 枚数（系统自动按 1:1 计算，不可手动填写）/ 启用状态 / 显示顺序
- **新增档位**：填写充值金额（IDR）→ 枚数自动计算（显示验证：N IDR = N 枚）→ 设置显示顺序 → 保存后对用户侧即时生效
- **启用 / 禁用**：运营可随时切换任意档位的启用状态；**规则：充值档位页至少须保留 1 个启用状态的档位**，系统不允许将全部档位同时禁用（操作时弹出阻断提示）
- **显示顺序**：支持拖拽排序；用户侧充值档位选择页按此顺序展示
- **操作权限**：仅限运营管理员
- **变更日志**：记录每次新增 / 禁用 / 顺序调整的操作人和时间，保留历史供审计查阅
- **上线初始档位已确定为 4 档：Rp10,000 / Rp25,000 / Rp50,000 / Rp100,000**（对应用户端 PRD A-14），运营团队须在上线前通过本配置页录入这 4 档并验证用户侧展示正常；上线后档位本身可随时通过本页增删调整，无需改代码

---

### AB-6C：PawCoin 浮存余额监控

对应用户端 PRD：FR-50A——平台须持续监控全平台 PawCoin 未消费余额总量（浮存），避免触及印尼闭环电子货币豁免门槛（约 Rp10 亿）。

**后果：**

- 入口：管理后台「PawCoin 管理」→「浮存余额监控」
- **实时概览**：
  - 全平台当前 PawCoin 未消费余额总量（单位：枚，等值 IDR，T+0 刷新）
  - 与法规豁免门槛（Rp10 亿）的距离，以进度条 + 百分比展示（如「已使用 23.4%，剩余 Rp7.66 亿空间」）
  - 本自然月净新增浮存（本月充值总量 - 本月消费总量）
- **趋势图**：近 30 天每日浮存余额变化折线图，辅助判断增长趋势
- **预警机制**：
  - 浮存超过门槛 **70%**：系统向运营管理员发送后台通知 + 邮件告警（黄色，提示"建议提前评估是否需要启动牌照申请"）
  - 浮存超过门槛 **90%**：红色高优先级告警（同渠道），提示"须立即评估是否暂停充值入口或启动正式牌照申请"
- **操作说明**：本视图仅供监控和预警，暂停充值入口须工程手动操作（V1.1.0 不提供后台一键暂停充值能力）；启动牌照申请须线下走合规流程，后台不直接支持

## 3. 模块 7：AI 问诊运营数据（V1.1.0 新增）

对应用户端 PRD：V1.1.0 FR-43C（红色评级安全护栏，超额使用监测）

**功能描述：** 红色评级安全提醒在任何情况下均免费展示（FR-43C 不可妥协护栏），但额度用尽后触发的红色评级记录为「超额使用」，供运营后台查看和分析，用于识别异常高频触发账号，辅助后续支付策略调整决策。

---

### AB-7A：红色评级超额使用监控视图

**后果：**

- 入口：管理后台「AI 问诊数据」→「红色评级超额使用」
- **列表视图**：展示触发过超额使用的用户列表，字段包括：
  - 用户账号 / 注册时间
  - 最近 7 天红色评级触发次数（含额度内 + 超额，分列展示）
  - 超额使用次数（本自然月）
  - 累计超额使用次数（账号生命周期）
  - 最近一次触发时间
  - 当前账号状态（正常 / 已标记待核查 / 已处理）
- **筛选与排序**：支持按时间范围、超额次数区间、账号状态筛选；默认按最近 7 天红色触发次数降序排列
- **单账号详情**：点击进入该用户的红色评级触发历史记录列表（每条：触发时间 / 危险等级 / 是否超额 / 额度状态），供运营人工核查时参考
- **标记与处理**：运营人员可对可疑账号手动标记「待核查」，记录核查备注（内部可见）；处理完成后标记「已处理」，历史标记记录不删除
- **不提供实时拦截能力**：本视图仅供人工运营查阅，不支持直接从后台封号、限制提交或触发自动响应——任何限制动作须走独立的账号管理流程（V1.1.0 暂未定义具体处罚机制，见 FR-43C OQ-12）
- **数据刷新频率**：`[NOTE FOR PM: 建议与工程确认数据是 T+1 批量更新还是准实时更新，对运营核查的时效性影响较大，建议至少 T+0 日内更新（当日触发当日可见）]`

## 4. 模块 8：收费运营数据（V1.1.0 新增）

对应用户端 PRD：FR-43A/43D/43E/49D（付费场景）、FR-50A（PawCoin 充值）

**功能描述：** V1.1.0 首次引入收费闭环，管理后台须支持定价动态配置、兽医咨询订单全流程查看、AI 问诊收入统计及兽医分成月结对账，供运营和财务日常管理使用。

---

### AB-8A：收费定价配置

对应用户端 PRD：FR-43A（AI问诊解锁）、FR-43D（兽医咨询）、FR-49D（身份证高清图下载）

**后果：**

- 配置入口：管理后台「收费运营」→「定价配置」
- **配置字段：**
  - AI 问诊详情解锁价格（IDR，当前值：Rp10,000）
  - 兽医咨询全价（IDR，当前值：Rp50,000，不区分是否由 AI 问诊结果页升级发起）
  - 身份证高清图下载价格（IDR，当前值：Rp5,000）
  - **兽医分成比例**（整数，单位 %，默认值 60）：兽医咨询场景下，兽医分得的比例，不区分基础发起或 AI 升级发起
- **生效时机**：保存后对新触发的支付弹层立即生效；已打开但尚未完成支付的弹层仍展示旧价格，不动态刷新（防止用户操作中途价格变化产生困惑）；分成比例修改仅影响修改后新完成的咨询，已完成咨询的分成按当时比例计算，不溯及调整
- **操作权限**：仅限运营管理员；价格修改须二次确认（弹窗展示变更前后对比，确认后保存）
- **变更日志**：记录每次修改的操作人、时间、各字段变更前后值，保留完整历史

---

### AB-8B：兽医咨询订单管理

对应用户端 PRD：FR-43D（兽医咨询收费）

**后果：**

- 入口：管理后台「收费运营」→「兽医咨询订单」
- **订单列表字段：**
  - 订单 ID / 用户账号 / 发起时间
  - 支付方式（QRIS / DANA / PawCoin）/ 支付金额（IDR）
  - 接诊兽医账号（已接单及以后阶段展示，等待中为空）
  - 订单状态：等待中 / 已完成 / 已取消 / 已暂停 / 退款申请中 / 已退款
- **筛选**：支持按时间范围、状态、支付方式、是否退款筛选；支持按用户账号或订单 ID 搜索
- **订单详情**：点击展开完整订单记录，含各阶段时间戳（支付成功 / Cancel-Void 结果 / 接单时间 / 完成时间 / 退款申请提交 / 退款处理完成）
- **PawCoin 支付订单说明**：PawCoin 支付的兽医咨询，月结时兽医分成仍以 IDR 结算（平台承担 PawCoin 与 IDR 的换算），订单详情展示「用户支付：X 枚 PawCoin（等值 Rp X）/ 兽医应得：Rp X」
- **不提供退款操作入口**：退款处理须通过客服 AB-5E 工单流程发起，本视图仅展示状态

### AB-8C：AI 问诊收入统计

对应用户端 PRD：FR-43A/43B（AI 问诊付费解锁）

**后果：**

  - AI 问诊提交总次数（含免费额度内）
  - 免费额度解锁次数 / 付费解锁次数
  - 付费转化率（额度用尽后实际付费比例，对应 SM-6）
  - 付费解锁收入合计（IDR + PawCoin 分别列示）
- **支持按日期范围自定义筛选**，数据精度 T+0 或 T+1（待工程确认）
- 设计目标：供产品团队追踪 SM-6 指标；V1.1.0 不做自动报表推送

### AB-8D：兽医分成月结对账

对应用户端 PRD：FR-43D（兽医咨询分成规则）

**后果：**

- 入口：管理后台「收费运营」→「月结对账」
- **月结单自动生成**：每月 1 日（次月 1 日 00:00 WIB，与 FR-43B 时区一致）自动生成上月所有兽医的分成对账单，生成后通知财务负责人
- **对账单内容（按兽医账号汇总）：**
  - 统计周期（自然月）
  - 完成咨询总笔数
  - 咨询分成：笔数 × 单价 × 兽医分成比例（按 AB-8A 当时生效配置计算，默认 60%）
  - 当月应付兽医金额（IDR）合计
  - `[NOTE: 分成基数统一以 IDR 计算，无论用户实际通过何种方式支付（QRIS/DANA/PawCoin），均按定价面值折算为 IDR 参与分成计算]`
- **全平台汇总行**：总完成咨询数 / 总分成支出（IDR）/ 总平台留存（IDR）
- **状态流转**：待财务确认 → 已付款（财务手动标记，必填付款凭证）→ 已归档
- **导出**：支持导出 CSV / Excel，供财务存档及汇款操作
- **历史月份**：可查看任意历史月份对账单；已归档月份数据只读，不可修改

### AB-8E：支付记录通用查询

### AB-8F：AI 问诊月免费额度配置

对应用户端 PRD：FR-43B（每月免费额度判定）

**后果：**

### AB-9B：工单详情与运营处置

**后果：**

**详情视图展示：**

- 待审核字段原文（大字展示）
- 用户账号基本信息（注册时间、历史违规处置次数）
- 三方风险评分及触发原因摘要（如「含引流关键词」，由三方服务返回）
- 历史处置记录（同账号曾被处理过的标识字段问题，按时间倒序列出）

**运营可执行的处置动作：**

| 动作 | 触发后果 |
|---|---|
| **不违规（放行）** | 工单关闭，内容维持已生效状态，不通知用户 |
| **要求修改** | 向用户发送系统通知「你的[昵称/宠物名]包含不当内容，请在 7 天内修改，否则账号功能将受限」；字段在其他用户界面替换为「***」展示（本人可见原内容），直到用户修改并通过审核 |

- 执行任何动作须填写处置备注（内部可见，用于记录判定依据）
- 处置完成后工单自动归档，计入该账号历史违规记录

### AB-9C：用户修改后的重新审核

**后果：**

- 用户在「要求修改」通知期限内提交新的昵称/宠物名后，内容再次进入异步审核流程（同 AB-9A 规则，重新评分）
- 新内容通过（风险评分 < 0.6 自动通过，或运营判定不违规）→「***」屏蔽自动解除，内容恢复正常展示；通知用户「你的[昵称/宠物名]已审核通过」
- 新内容再次被标记高/中风险 → 重新进入审核队列，工单关联原处置记录，运营可查看修改历史
- **逾期未修改处理**：超过 7 天（运营可在 AB-9B 处置时修改期限）用户仍未提交修改 → 系统自动限制该账号功能（不可发帖、不可评论，可正常浏览），并再次推送提示通知；账号功能限制在用户完成修改并通过审核后自动解除

## 6. V1.0.0 管理后台行为变更（V1.1.0 起生效，覆盖移交版）

> **背景说明：** V1.0.0 管理后台 PRD 已提前移交技术团队开发。以下条目是在 V1.1.0 需求确认阶段对 V1.0.0 既有后台行为的修正与补充，**技术实现须以本节为准，覆盖移交版中的对应描述**。

### 6.1 AB-2E 兽医封禁——付费会话处理（覆盖移交版）

**移交版行为（不实现）：** 兽医被封禁时，付费咨询会话强制关闭；App 向用户弹出「重新匹配」或「申请退款」选项。

**V1.1.0 实际行为：**

- 封禁操作触发时，系统检测该兽医是否有进行中的付费会话
- 有进行中付费会话 → 后台显示提示「该兽医有进行中的付费咨询，将在会话结束后自动完成封禁」，运营确认后操作进入**挂起状态**
- 会话自然结束后，系统自动触发封禁生效；用户侧会话体验完整，不发送异常通知，不退款
- 无进行中付费会话 → 封禁立即生效（与移交版一致）
- 与 AB-2H SIPDH 到期、AB-UA-02 用户停用的挂起机制设计一致，可复用同一套「挂起-监听会话结束-自动触发」逻辑

**联动影响：** AB-4A 异常工单仅在**免费咨询**（V1.0.0）会话被强制关闭时触发；付费会话因不再强制关闭，不生成异常工单。

### 6.2 AB-2H SIPDH 到期——进行中会话处理（补充移交版）

**移交版行为（不完整）：** 移交版 AB-2H 未说明 SIPDH 到期当日若有进行中会话时的处理逻辑。

**V1.1.0 补充规则：**
- SIPDH 到期当日，若该兽医有进行中的会话，**等待会话自然结束后再执行状态切换**，不强制中断已建立的会话
- 实现方式与 AB-2E 挂起机制一致（可复用）

### 6.3 AB-UA-02 用户停用——付费会话处理（补充移交版）

**移交版行为（不完整）：** 移交版 AB-UA-02 停用用户时「进行中的问诊会话强制关闭」。

**V1.1.0 补充规则（付费咨询阶段）：**
- 停用操作触发时，系统检测该用户是否有进行中的付费兽医咨询（已接单、会话建立中）
- 有进行中付费会话 → 后台显示提示「该用户有进行中的付费咨询，将在会话结束后自动完成停用」，运营确认后操作进入挂起状态；会话结束时系统自动触发停用，**不退款**（用户违规导致停用，已交付服务不补偿）
- 免费咨询阶段行为与移交版一致（强制关闭）

### 6.4 AB-3A 举报处理——回告行为澄清（补充移交版）

**移交版内容（部分不准确）：** 移交版 AB-3A 包含了举报处理后向举报人发送通知的描述，该行为实际属于 **V1.1.0 新增功能**（FR-51），V1.0.0 不实现回告。

**V1.1.0 澄清：**
- **V1.0.0**：举报处理后不向举报人发送任何通知
- **V1.1.0 起**：运营标记处理完成后，系统自动回告举报人（见本文档 §5 及用户端 FR-51）
- 移交版 AB-3A 中包含通知逻辑的部分在 V1.0.0 阶段**不实现**，对应工程工作量归入 V1.1.0 排期
