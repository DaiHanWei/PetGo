# 广告位方案 v1.0.0（四版本对比）

**状态：** 待老板拍板选择版本
**记录日期：** 2026-07-13
**前置说明：** 基于《广告位方案（探索讨论）》的探索结论——品牌直客模式，聚焦宠物食品/日用品类目；已明确广告位落在底部 tab 根页面层级。本文档给出四个可选实现版本的具体设计，供选择：版本 A/B 在首页 Feed（四个底部 tab 中唯一游客可访问、不需要登录的一个），版本 C 落在「问诊」tab（Konsultasi Kilat 问诊 hub，登录用户可见），版本 D 是内容植入，形式和落点都与前三版性质不同，见 §4。

## 1. 版本 A：原生信息流广告

### 1.1 视觉设计

比照现有 `MasonryCard`（内容卡片）的结构做同款外壳，让广告卡"看起来像"一条普通帖子：

- **卡片容器**：同样的圆角 + 描边 + 背景色，与内容卡片外观一致
- **头部行**：用品牌 Logo 代替用户头像、品牌名代替昵称；原本放"发布时间"的位置，固定显示"Disponsori"/「广告」字样（不用相对时间，因为这本来就不是一条真实动态）
- **类型徽章位**：内容卡片这个位置放的是 Momen / Tips / Cerita 三色徽章，广告卡这里换成一个显眼但独立配色的「Iklan」徽章——**这个徽章必须做得和三个内容类型徽章视觉上明显不同色系**，不能让用户以为广告是第四种内容分类
- **正文**：品牌短文案，最多两行，和内容卡片文字规格一致
- **图片**：全宽 4:3 图，和内容卡片一致的裁切比例

### 1.2 插入位置与频率

在 `feed_view.dart` 渲染卡片列表时，由客户端按固定间隔（如每 8 条内容）在列表中插入一条广告卡——**这是纯客户端本地行为，不需要后端 Feed 分页接口感知广告的存在**，后端游标分页（`nextCursor`/`hasMore`）完全不受影响。好处：不用碰现有 `FeedItem`/内容类型（`DAILY`/`GROWTH_MOMENT`/`KNOWLEDGE`）数据模型，不用改后端接口。

### 1.3 工程量评估

中等偏低——比"给 FeedItem 新增 AD 类型、后端也要感知"这种做法轻很多，因为广告卡是客户端插入的独立组件，不复用 `FeedItem` 数据结构。主要工作量在新建一个广告卡组件 + 客户端插入逻辑。

### 1.4 风险/顾虑

- 需要克制"像不像"的度——如果做得太像真实内容，容易引发"平台伪装广告"的投诉和监管风险（探索阶段已提到"分不清内容还是广告"的产品调性顾虑）；「Iklan」标注必须清晰，不能为了融入感而弱化标注
- 每隔 N 条插入，意味着刷得越多看到的广告越多，曝光和用户浏览深度绑定，不是每次打开 App 都保证曝光一次

## 2. 版本 B：首页顶部 Banner

### 2.1 视觉设计

- **位置**：首页头部组件内，分类 Tab（全部/日常/成长时刻/知识）下方、瀑布流卡片上方
- **尺寸**：矮横幅比例（如 3:1），与页面其他卡片同样的左右留白，不做满屏出血
- **标注**：右上角「广告」小标签，半透明徽章
- **交互**：整条可点击，App 内 WebView 打开品牌落地页
- **无生效素材时**：这一区块直接不渲染，不留空白

### 2.2 插入位置与频率

固定插入一次，不随内容穿插，不受滑动深度影响——只要用户打开或刷新首页，必定看到一次（除非后台没有配置生效中的素材）。

### 2.3 工程量评估

两版本里最低——在已有的首页头部组件里加一项，不涉及 Feed 列表渲染逻辑，不需要设计"插入频率"这类客户端逻辑。

### 2.4 风险/顾虑

- 视觉上更"广告感"，一眼就能看出不是内容，对社区调性冲击比版本 A 更直接
- 用户"一进 App 就是广告"的观感冲击集中在开屏那一刻，而不是像版本 A 分散在滑动过程中

## 3. 版本 C：问诊页（Konsultasi Kilat Hub）Banner

### 3.1 位置

问诊 tab 根页（`triage_page.dart`）从上到下结构是：Hero 文案 → **AI 分诊卡 + 兽医咨询卡（平级双入口，页面核心转化点）** → 进行中会话卡（若有）→ 我的问诊历史列表。

Banner 插在**两个入口卡下方、进行中会话卡/问诊历史列表之前**——不插在两卡中间，避免打断这对"平级双入口"精心设计的视觉动线；也不放在两卡之上，避免一进页面先看到广告而不是核心功能入口。

### 3.2 视觉设计

沿用版本 B 的横幅样式（矮横幅比例、右上角「广告」标注、整条可点击、无生效素材时不渲染），保持两个 Banner 位视觉语言一致，用户认知成本低。

**关于文案**：可以做成贴合场景的软性文案（例如围绕"驱虫/营养"关键词），但探索阶段文档 §1 已提到印尼对宠物食品功效宣称可能有本地监管要求——**MVP 阶段建议先用通用品牌曝光文案，不做功效关联表述，避免未经法务确认就上线场景化宣称**。

### 3.3 工程量评估

与版本 B 相当——在 `triage_page.dart` 的 `ListView` children 里加一个条件渲染 Widget，插入点单一，不涉及问诊/分诊的业务逻辑。

### 3.4 曝光范围与商业价值

- 问诊 tab 需要登录才能进入（游客点击会先被登录墙拦截），**曝光范围只有登录用户**，比首页 Feed 的两个版本窄
- 但用户进入这个 tab 的意图非常聚焦——就是在处理"宠物健康"相关的事，与广告主聚焦类目（宠物食品/驱虫/日用品）的场景关联度天然比首页 Feed 更高，单次曝光的转化质量可能更高，即使总曝光量更小
- 这本质上是文档 §3 候选表里"AI 问诊结果页"精准场景优势的一个更安全的变体：落在问诊 hub 入口页而不是诊断结果页，此时用户还没拿到具体的绿/黄/红分级结果，不涉及"在诊断结论旁边卖驱虫产品"这种更敏感的关联，合规和观感风险都比直接放结果页低

### 3.5 风险/顾虑

- 这个页面的核心目的是让用户使用 AI 分诊/兽医咨询服务，插入商业内容容易被解读为"用户来求助宠物健康问题，平台却在这里卖广告"，观感上比首页 Feed 更需要克制，标注和位置都要做得足够清楚
- 场景化文案的功效宣称风险（见 3.2），上线前需要法务过一遍

## 4. 版本 D：内容植入

广告主品牌以「文章/图文/短视频」形式出现在内容区，标注「广告」或「赞助」——这个版本和 A/B/C 性质不一样：A/B/C 都是"广告位"（固定坑位插一个配置化的素材），D 是"给品牌开一个内容发布通道"，本质是内容能力，不是广告位能力。先说清楚这个区别，再决定要不要做，因为工程量和运营模式都完全不同。

### 4.1 呈现形式

品牌发布的内容和普通用户帖子一样，走真实的内容详情页（`content_detail_page.dart`），不需要为它单独设计详情页——因为它本来就是一条真实内容，只是作者是品牌方。在 Feed 卡片和详情页的作者信息位置，加一个「广告」/「赞助」标签，和当前 `MasonryCard` 已有的 Momen/Tips/Cerita 类型徽章并列或替换显示。

### 4.2 出现位置

不像 A/B/C 有固定坑位——内容植入的帖子就是一条 `FeedItem`，混在真实内容里，由 Feed 现有的排序逻辑（时间/推荐）决定它出现在哪，不需要客户端做"每隔 N 条插入"这类特殊逻辑。

### 4.3 工程量评估——比 A/B/C 都重，且有一块能力目前完全空白

- 需要在 `FeedItem` 数据模型上加一个"赞助标记"字段（后端 + 前端都要改），这是 A/B/C 都刻意避开的改动，D 绕不开
- `MasonryCard` 的徽章渲染逻辑要改，支持"赞助"这个新标签和已有三个内容类型徽章共存
- **"短视频"这个形式，App 里现在完全没有**——查过代码，`media` 模块目前只处理图片上传（OSS 图片上传），没有视频上传、没有视频播放器，`ContentType` 也只有 `DAILY`/`GROWTH_MOMENT`/`KNOWLEDGE` 三种图文形态。要支持短视频植入，等于要先把"视频内容"这个能力在 App 里从零建起来——这不是广告位的工作量，是一个独立的产品功能，需要单列评估、单独排期，不能算进这次广告位的范围里
- 建议 MVP 阶段先只做「文章/图文」两种形式（复用现有 `KNOWLEDGE`/`DAILY` 内容形态和图片上传能力），短视频留到 App 本身支持视频内容之后再考虑

### 4.4 内容从哪来——不是"上传一张素材图"，是"发一条真实内容"

A/B/C 的后台需求是一张简单表单（图片+链接+时间）。D 不是这个量级，品牌方要提供的是一篇真实文章或图文内容，最简单的落地方式：

- 不给品牌方开真实 App 账号走注册登录，而是由内部运营在后台代品牌方创建内容（标题、正文、配图、赞助标记开关），复用现有内容审核（moderation）模块的发布能力
- 审核维度也要比普通内容审核多一层——不只是"内容合不合规"，还要审"调性是否贴近社区"、"是否涉及功效宣称"（驱虫/营养品类目，探索阶段文档 §1 已提示的合规点）

### 4.5 优点——这是唯一真正回应了"内容可控性强"这个品牌直客优势的版本

探索阶段文档 §0 提到品牌直客模式相对程序化广告网络的优势是"内容可控性强，可以要求广告形式贴近社区调性"——A/B/C 本质上还是"广告位"，只是形式做得像内容；D 才是真正意义上的"品牌内容化"，用户体验上最接近文档设想的"内容化投放，如喂养科普向内容"这个理想状态，长期来看观感和信任成本最低。代价是这次是四个版本里工程量最高、且依赖"内容"而非"广告位"这套更简单的机制。

## 5. 共用的后台配置设计（适用于版本 A / B / C）

版本 D 走的是独立的内容创建 + 审核流程（见 §4.4），不适用本节。以下只针对 A/B/C 这三个"广告位"版本：

无论选 A、B、C 哪个，后台管理页面的核心字段一致（不做多广告主轮播，每个广告位 MVP 阶段只支持一条生效中的素材）：

- banner/广告卡图片上传
- 跳转链接
- 生效开始/结束时间
- 启用开关
- （仅版本 A 需要）插入间隔 N，MVP 阶段可先写死默认值，不做成运营可调的复杂参数

**若同时上线不止一个广告位**（例如首页 + 问诊页两个都要）：需要在配置里加一个「位置」字段区分是哪个广告位的素材，本质还是同一张简单表单，多一个下拉选择，不改变整体复杂度。

后台可复用现有 admin 模块的写法（与内容审核、用户管理模块同构），新增一个简单 controller + 表单页即可，不用为此单独建一套广告位管理系统。

## 6. 视觉规范细化（复用现有 Design Token，不新造一套皮肤）

四个版本全部复用 App 现有的设计 token（`AppColors`/`AppTypography`/`AppRounded`/`AppSpacing`/`AppShadows`），不引入新的色板或字阶——广告单元要"看起来是这个 App 里的东西"，而不是贴牌进来的外来组件。

### 6.1 「广告/赞助」标签——两种呈现方式，别混用

App 现有三个内容类型徽章（`MasonryCard` 里的 Momen/Tips/Cerita）都是"浅色底 + 深色饱和文字"的 pastel pill：绿底绿字、黄底黄字、紫底紫字。广告标签必须让用户一眼识别出"这不是第四种内容分类"，同时还要在不同背景下都保持清晰——这两个约束对应两种不同的实现，不能用同一套参数应付所有位置：

| 场景 | 标签所在背景 | 处理方式 |
| --- | --- | --- |
| **版本 A（原生信息流卡）、版本 D（内容植入）** | 卡片头部行，纯白 `AppColors.card` 背景，和三个内容徽章同一个槽位 | 沿用 pastel pill 的形状规格（`BorderRadius.circular(6)`、`AppTypography.badge` 10sp/w600），但改用**中性灰色系**：底色 `AppColors.line2 (#F0F0F0)`、文字 `AppColors.ink2 (#544864)`。灰色系不在绿/黄/紫任何一个内容徽章色系里，一眼可辨"这不是内容分类"，同时因为背景固定是白色，对比度稳定可预先验证达标 |
| **版本 B（首页顶部）、版本 C（问诊页）** | 叠加在品牌素材图片上，图片颜色/明暗不可控 | 不能用底色固定的 pastel pill（万一素材图也是浅色，pill 会糊在图上看不清）——改用**深色半透明贴纸**：`rgba(0,0,0,0.55)` 底 + 白字，位置固定在图片右上角，内边距 8px。半透明深底不管素材图明暗都能保证对比度，这是图片广告类产品的通用做法 |

文案统一用「广告」二字（中文语境）或「Iklan」（印尼语境，视 App 当前语言设置），不用"赞助""Disponsori"这类更软的措辞跟"广告"混着用——同一个 App 里标签措辞必须统一，不然用户会怀疑是不是有两种不同性质的商业内容。

### 6.2 版本 A / D 共用：卡片外壳

100% 复用 `MasonryCard` 现有规格，不新建视觉语言：

- 容器：`AppRounded.phoneRadius`（14px 圆角）+ `Border.all(color: AppColors.border)` + `AppColors.surface` 背景，和内容卡片零差异
- 头部行：头像位置放品牌 Logo（同样走 `LetterAvatar` 的圆形裁切规格，或品牌方提供已裁好的圆形 Logo）；昵称位置放品牌名，用 `AppTypography` 里同名字号（13.5sp/w700, `AppColors.ink`）；原本放"发布时间"的位置改放固定文案「广告」（不放相对时间——broadcast 类素材没有"刚刚发布"这个语义）
- 正文/图片：字号、行高、4:3 裁切比例，和内容卡片完全一致（`AppTypography.body`，2 行截断）

### 6.3 版本 B / C 共用：Banner 容器

- 圆角：`AppRounded.phoneRadius`（14px），和内容卡片同一圆角家族——**不要**照搬 `_GuestJoinBanner` 那种 20px 圆角+强阴影的"弹窗感"处理，那是引导登录的强 CTA 语言，广告位要低调，不能抢内容的视觉权重
- 阴影：`AppShadows.sm`（最轻量的一档），不用 `md`/`lg`，避免广告位比内容卡片本身更"跳"
- 边框：`Border.all(color: AppColors.border)`，和内容卡片一致
- 左右留白：`AppSpacing.screenEdge`（16px），和页面其他内容对齐，不做满宽出血
- 尺寸：横向占满内容宽度、纵横比 3:1（约 343×114dp，取决于机型宽度）——高度远超 44pt 触达下限，不需要额外扩大点击热区
- 触达反馈：整条 `GestureDetector`（和 `MasonryCard` 现有点击方式一致，App 里内容卡片本身也没有按压态涟漪反馈，广告位保持和内容卡片同样的交互手感，不单独引入涟漪，避免广告位反而比内容卡片"更精致"）

### 6.4 无障碍检查清单（对照 ui-ux-pro-max Quick Reference §1/§2 逐项过）

- **色彩不是唯一信息载体**：「广告」二字本身就是文字标签，不是靠颜色区分——已满足 `color-not-only`，不依赖用户能分辨灰色 pill 或深色贴纸的"颜色含义"
- **对比度**：6.1 里灰色 pastel pill 用的 `ink2 on line2`，和 App 现有 `textSecondary` 在浅底上的用法同源，沿用已验证过 NFR-13 AA ≥4.5:1 的组合，不用重新测；深色贴纸用纯白字 on 55% 黑，属于高对比区间，两种方案都不会有对比度问题
- **语义标签（读屏器）**：广告卡片/Banner 需要包一层 `Semantics(label: "广告：<品牌名>")`，和 `MasonryCard` 现有的 `Semantics(button:, label:)` 包法一致，让 VoiceOver/TalkBack 明确读出这是广告，不是普通内容
- **触达尺寸**：Banner（B/C）整条可点，热区远超 44×44pt；卡片（A/D）复用 `MasonryCard` 现有点击区域，同样不是问题
- **动效**：不新增任何进场动画——Feed 现在没有逐条卡片入场动画，广告单元也不应该单独加，免得广告位反而比内容"更花哨"，这是 `excessive-motion`/`motion-meaning` 明确要避免的"纯装饰性动效"
